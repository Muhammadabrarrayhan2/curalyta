# Neon + Cloudflare R2 + Render + Vercel Deployment Design

**Problem**

The current deployment design assumes Supabase for both Postgres and private document storage, but the team is constrained by the `2 active free projects` limit and needs capacity for other projects as well.

The current codebase also still assumes:
- PostgreSQL via a single `DATABASE_URL`
- local disk storage for uploaded documents
- a simple relative `/api` frontend model for all backend traffic

That combination is not yet safe for a production deployment on:
- frontend on Vercel
- backend API on Render
- PostgreSQL on Neon
- private medical document storage on Cloudflare R2

**Goals**

- Deploy the frontend to Vercel using the existing `frontend` app
- Deploy the backend API to Render using the existing `backend` app
- Move production relational data to Neon Postgres
- Move uploaded medical documents to a private Cloudflare R2 bucket
- Preserve the existing relative `/api` frontend contract for normal JSON API calls
- Keep document access private and enforced by backend authorization checks
- Reduce deployment risk by using additive schema changes and safe fallbacks
- Avoid committing secrets or exposing server-only credentials in frontend bundles

**Non-Goals**

- Moving authentication to a managed auth provider
- Making document buckets public
- Letting the browser upload directly to R2 in the first rollout
- Adding custom production domains in the first rollout
- Refactoring unrelated product features during deployment work

**Recommendation**

Use Vercel external rewrites so the frontend continues calling `/api` for normal API traffic while Vercel proxies those requests to the Render backend.

Do not force document upload and download binaries through that same path in the first rollout. Keep JSON API traffic on the existing relative `/api` path, but handle document binary flows through the backend origin directly after authorization checks. This reduces the chance of proxy-related size, timeout, or body handling issues during the migration.

Use Neon Postgres for production data and Cloudflare R2 for private object storage. Keep authorization centralized in the backend. For downloads, prefer short-lived signed R2 URLs generated by the backend after access validation instead of streaming every file through Render.

**Design**

## 1. Platform topology

The deployed system will use one GitHub repository and four hosted services:

- GitHub repository for source control and deployment source
- Vercel project for `frontend`
- Render web service for `backend`
- Neon project for PostgreSQL
- Cloudflare R2 bucket for private medical documents

The frontend remains a static SPA built from the `frontend` workspace. The backend remains an Express API built from the `backend` workspace. Neon stores production relational data. R2 stores production document objects.

The first rollout will use platform default hostnames:
- frontend: `*.vercel.app`
- backend: `*.onrender.com`
- database: Neon-managed hostname
- storage: Cloudflare R2 S3-compatible endpoint

## 2. Frontend routing and API strategy

The frontend should preserve the current relative `/api` base URL for standard JSON API traffic because the app already assumes that pattern.

The Vercel project should therefore include a `vercel.json` configuration that:
- rewrites `/api/:path*` to the Render backend origin
- rewrites SPA deep links to `index.html`

This keeps most frontend behavior unchanged and minimizes regression risk.

However, the document module should not assume that all file uploads and downloads can safely pass through the same Vercel rewrite path. Binary file traffic is the highest-risk part of the deployment because the current backend expects multipart uploads and local disk downloads.

For the first rollout:
- standard API requests continue through `/api`
- document upload requests should target the backend origin directly
- document download initiation should target the backend origin directly
- the backend should return a short-lived signed R2 URL or redirect for authorized downloads

The frontend can support this with a second safe browser env variable for the public backend origin used only by the document client, for example `VITE_BACKEND_ORIGIN`. This is safe to expose because it is only a public service hostname, not a secret.

## 3. Backend deployment on Render

The Render service will target the `backend` directory as its root directory. It will run as a public web service with:

- dependency install and TypeScript build during deploy
- a start command using the compiled server entrypoint
- a health check path using `/api/health`

The backend must remain the only component that knows:
- database credentials
- R2 access credentials
- JWT secret
- AI provider credentials when AI features remain enabled in production

Backend CORS must explicitly allow:
- the Vercel production URL
- Vercel preview URLs if preview deployments are enabled
- local development origins

The backend startup path must remain Render-safe by respecting the injected `PORT` value and binding reliably in hosted environments.

## 4. Production database on Neon Postgres

Production relational data will move from local or Docker Postgres to Neon Postgres.

Use two database connection strings:

- `DATABASE_URL` for application runtime
- `DIRECT_URL` for Prisma migrations and admin CLI usage

This matches the safest Prisma-on-managed-Postgres pattern for hosted environments:
- pooled runtime connections for the application
- direct connections for schema migration operations

Prisma configuration should be updated so the datasource can use both URLs without changing the application-level Prisma client usage.

The first-rollout production migration strategy is:

1. provision the Neon project and production database
2. configure Render environment variables
3. deploy the backend
4. run `prisma migrate deploy` against production using `DIRECT_URL`
5. verify health checks and core CRUD behavior

## 5. Private medical document storage on Cloudflare R2

The current document module writes files to local disk using `multer.diskStorage`. That is not suitable for Render because local filesystem changes are not durable across deploys or instance replacement.

Document storage will move to a private R2 bucket, for example `medical-documents`.

The upload flow will be:

1. authenticated browser sends multipart upload to the backend
2. backend validates role, patient ownership, file type, and file size
3. backend uploads the file buffer to R2 using server-side credentials
4. backend stores document metadata and object location in Neon

The download flow will be:

1. authenticated browser requests download from the backend
2. backend validates access rights for doctor, patient, or admin
3. backend generates a short-lived signed R2 URL or issues a redirect to it
4. browser downloads the object directly from R2

The delete flow will be:

1. authenticated doctor requests deletion
2. backend verifies ownership
3. backend deletes the object from R2
4. backend deletes the metadata row from Neon

This order is safer than deleting the metadata row first because it reduces the chance of leaving an undeclared orphaned object in storage.

## 6. Safe migration shape for the document model

The current `Document` model stores a local filesystem `path`. Changing production storage without care could break access to legacy records or make rollback difficult.

The migration should therefore be additive:

- keep the existing `path` field temporarily for backward compatibility
- add new storage metadata fields such as:
  - `storageProvider`
  - `storageBucket`
  - `storageKey`

During the transition period:
- records with `storageProvider = R2` use signed-URL download behavior
- legacy records without that metadata can still resolve through the local-path fallback in development or controlled migration scenarios

This additive approach is safer than replacing `path` immediately because it supports staged rollout, rollback, and mixed-state validation.

## 7. Storage abstraction and fallback behavior

Do not hardcode R2 logic directly into the route handlers. Introduce a storage abstraction so the rest of the documents module talks to one interface regardless of provider.

The backend should expose one document storage service with operations such as:
- save uploaded object
- produce authorized download access
- delete stored object

Provide at least two implementations:
- local filesystem storage for development and fallback
- R2 storage for hosted production

This keeps the authorization and route logic stable while changing only the storage backend, which reduces regression risk.

## 8. Secret management and repository hygiene

The repository must never contain real production secrets.

Required backend-only secrets include:
- `JWT_SECRET`
- `DATABASE_URL`
- `DIRECT_URL`
- `R2_ACCOUNT_ID`
- `R2_ACCESS_KEY_ID`
- `R2_SECRET_ACCESS_KEY`
- `R2_BUCKET`
- `R2_ENDPOINT`
- AI provider keys when applicable

Safe browser-side variables may include:
- `VITE_BACKEND_ORIGIN` for document-specific direct backend requests

The repository should include:
- safe example environment files
- updated deployment documentation
- `.gitignore` coverage for local env files and generated secrets

The first rollout should continue using manual dashboard configuration across GitHub, Vercel, Render, Neon, and Cloudflare rather than introducing infrastructure-as-code in the same change set.

## 9. Rollout sequence

The recommended rollout order is:

1. prepare repository hygiene and example environment files
2. add additive Prisma schema changes for document storage metadata
3. introduce the document storage abstraction with local storage still as the default
4. verify upload, download, and delete behavior locally using the local provider
5. add the R2 provider implementation and production env parsing
6. update frontend document requests to use the backend origin where required
7. deploy the backend to Render with Neon and R2 credentials configured
8. run production migrations using `DIRECT_URL`
9. verify backend health and document flows in the hosted environment
10. deploy the frontend to Vercel with API rewrites and document origin config
11. verify SPA routing, auth, CRUD flows, and document access behavior

## 10. Validation

Deployment is considered successful when all of the following are true:

- frontend loads correctly from `*.vercel.app`
- frontend deep links work after refresh
- normal frontend `/api/*` calls succeed through the Vercel rewrite to Render
- Render backend passes `/api/health`
- backend reads and writes production data in Neon
- document upload stores objects in the private R2 bucket
- authorized doctors can upload, list, and delete documents
- authorized patients can access only their own documents
- unauthorized users receive access denial
- document downloads use short-lived signed access rather than exposing public files
- no production secret is committed to Git or exposed in frontend assets

Validation commands before rollout:

- `npm run build --workspace=backend`
- `npm run typecheck --workspace=backend`
- `npm run build --workspace=frontend`
- `npm run typecheck --workspace=frontend`

Manual validation after hosted deployment:

- doctor upload document flow
- doctor list documents flow
- patient document access flow
- admin access flow
- unauthorized access denial
- document delete flow
- signed URL expiry behavior

**Files Expected To Change**

- `frontend/vercel.json`
- `frontend/src/lib/api.ts` or a new document-specific API helper
- `backend/.env.example`
- `backend/prisma/schema.prisma`
- `backend/src/config/index.ts`
- `backend/src/lib/prisma.ts`
- `backend/src/modules/documents/documents.module.ts`
- one or more new backend storage service files for local and R2 providers
- `README.md`

**Risks and Mitigations**

- Render local disk is non-durable
  - Mitigation: move production document persistence to R2
- Binary uploads proxied through the wrong layer may hit body-size or timeout issues
  - Mitigation: keep standard JSON API traffic on Vercel rewrite but route document binary traffic to the backend origin directly
- Prisma migrations against pooled managed Postgres connections can fail
  - Mitigation: separate runtime `DATABASE_URL` from migration `DIRECT_URL`
- Storage migration can break legacy records if the schema change is destructive
  - Mitigation: use additive fields and keep temporary local-path compatibility
- Misconfigured CORS can block Vercel or local frontend traffic
  - Mitigation: make hosted and local origins explicit in config and documentation
- Secret leakage through repo or client bundle
  - Mitigation: keep R2 and database credentials server-only and verify ignored files before push
- Delete failures can leave object and metadata state inconsistent
  - Mitigation: delete the R2 object first, log failures clearly, and only remove metadata after storage deletion succeeds

**Implementation Note**

The safest implementation path is to keep behavior stable for non-document API routes and isolate the migration surface to:
- configuration parsing
- Prisma document metadata
- document storage adapter logic
- document-specific frontend request handling

That minimizes the number of moving parts changed at once and reduces the chance of introducing cross-cutting regressions.
